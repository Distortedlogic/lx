# Goal

Merge `PatternVisitor` and `TypeVisitor` into a single `AstVisitor` trait, replace the hand-written walk and dispatch functions with declarative macro invocations that leverage the uniform named-struct variants (from Phase 1), standardize where `leave_X` is called, and add a `visitor::prelude` module. This eliminates ~700 lines of hand-written boilerplate and ensures the walk infrastructure stays in sync with the AST automatically.

# Why

- Every visitor implementation must write `impl PatternVisitor for Foo {}` and `impl TypeVisitor for Foo {}` even when it has no pattern or type hooks — pure boilerplate visible in every lint rule and the CoreValidator
- The ~40 hand-written walk functions across four files all follow the same pattern (`node.walk_children(v, arena)?; v.leave_X(...)`) but are maintained manually — they fall out of sync when AST variants are added
- `leave_expr` is called inside `walk_expr` (line 218) while `leave_stmt` is called inside `dispatch_stmt` (line 101) — inconsistent and error-prone for anyone writing custom walks
- Every visitor consumer needs 5-8 imports (`AstVisitor`, `PatternVisitor`, `TypeVisitor`, `VisitAction`, `dispatch_stmt`, etc.) — a prelude eliminates this friction

# What changes

1. Move all methods from `PatternVisitor` and `TypeVisitor` into `AstVisitor` with their existing default implementations. Delete `PatternVisitor`, `TypeVisitor`, their files, and the supertrait bounds on `AstVisitor`
2. Remove all `impl PatternVisitor for X {}` and `impl TypeVisitor for X {}` blocks from every visitor consumer. For `capture.rs`, the non-empty `impl PatternVisitor for FreeVarCollector` methods must be moved into the `impl AstVisitor for FreeVarCollector` block, not deleted
3. Add a `define_walk_and_dispatch!` declarative macro that generates both the walk function and dispatch function for a given variant (walk calls `walk_children` + leave hook; dispatch calls visit hook + checks VisitAction + delegates to walk). Invoke it for every Expr variant. Pattern and TypeExpr sub-variant dispatchers have non-standard signatures and must remain hand-written
4. Standardize `leave_X` placement: the dispatch function always calls leave, the walk function only traverses children. Apply this uniformly to all four node kinds
5. Create `visitor/prelude.rs` re-exporting `AstVisitor`, `VisitAction`, `walk_program`, `dispatch_stmt`, `dispatch_expr`, and common AST types
6. Delete `visitor_pattern_hooks.rs`, `visitor_type_hooks.rs`, and collapse `walk_expr.rs` + `walk_expr2.rs` into `walk/generated.rs` containing macro invocations. Keep `walk_pattern.rs` and `walk_type.rs` hand-written but simplified

# How it works

After Phase 1, every Expr variant holds a named struct with a `walk_children` method. The `define_walk_and_dispatch!` macro generates a walk function and dispatch function per variant:

The walk function calls `node.walk_children(v, arena)` to recurse, nothing else. The dispatch function calls the visit hook, checks VisitAction (Stop/Skip/Descend), calls walk on Descend, and calls the leave hook after walk returns (or immediately on Skip). This standardizes the leave-call location across all node kinds.

For Expr walks, all variants use `walk_dispatch_id!` after Phase 1 (the `walk_dispatch_id_slice!` macro is no longer needed since all Expr variants are struct-wrapped). The `define_walk_and_dispatch!` macro replaces `walk_dispatch_id!` by generating both the dispatch and walk functions together.

For Pattern and TypeExpr sub-variant dispatchers, every single one has a non-standard signature — they destructure enum fields and pass them individually rather than passing a struct reference. These cannot be generated by the proposed macro. Specifically:

Pattern dispatchers with non-standard signatures:
- `walk_pattern_tuple_dispatch(v, id, elems, span, arena)` — takes `&[PatternId]` instead of struct ref
- `walk_pattern_list_dispatch(v, id, elems, rest, span, arena)` — takes `&[PatternId]` + `Option<Sym>`
- `walk_pattern_record_dispatch(v, id, fields, rest, span, arena)` — takes `&[FieldPattern]` + `Option<Sym>`
- `walk_pattern_constructor_dispatch(v, id, name, args, span, arena)` — takes `Sym` + `&[PatternId]`

TypeExpr dispatchers with non-standard signatures:
- `walk_type_applied_dispatch(v, id, name, args, span, arena)` — takes `Sym` + `&[TypeExprId]`
- `walk_type_list_dispatch(v, id, inner, span, arena)` — takes `TypeExprId`
- `walk_type_map_dispatch(v, id, key, value, span, arena)` — takes `TypeExprId` + `TypeExprId`
- `walk_type_record_dispatch(v, id, fields, span, arena)` — takes `&[TypeField]`
- `walk_type_tuple_dispatch(v, id, elems, span, arena)` — takes `&[TypeExprId]`
- `walk_type_func_dispatch(v, id, param, ret, span, arena)` — takes `TypeExprId` + `TypeExprId`
- `walk_type_fallible_dispatch(v, id, ok, err, span, arena)` — takes `TypeExprId` + `TypeExprId`

Approach: generate Expr walk/dispatch functions (all uniform after Phase 1), keep Pattern and TypeExpr walk/dispatch functions hand-written but simplified (using `walk_children` where possible).
