# REF: OpenAI Harness Engineering & Long-Running Agent Guides

Consolidated extraction from OpenAI's key harness and agent engineering publications (2025-2026).

## Source 1: Harness Engineering — Leveraging Codex in an Agent-First World

URL: https://openai.com/index/harness-engineering/

### The Experiment

3 engineers built ~1M lines of production code in 5 months. Zero hand-written code. ~1,500 merged PRs. 3.5 PRs per engineer daily. Throughput increased (not plateaued) when team expanded to 7. Agents performed 6+ hour unattended runs on single tasks.

### Core Methodology

Humans steer, agents execute. Engineers never wrote code directly. When failures occurred, the response was always identifying missing capabilities, not manual fixes.

### Three Harness Components

**1. Context Engineering**
- Repository-local documentation (architecture, APIs, style guides)
- AGENTS.md files encoding project rules (~100 lines, serving as table of contents)
- Cross-linked design documents with linter validation
- Agent access to observability data (logs, metrics, traces) and browser navigation
- "Give Codex a map, not a 1,000-page instruction manual"

**2. Architectural Constraints**
Rigid layered architecture: Types → Config → Repo → Service → Runtime → UI

Cross-cutting concerns (auth, telemetry, feature flags) enter through single explicit interfaces called Providers.

Enforced mechanically via:
- Custom linters (generated by Codex itself)
- Structural tests validating dependency directions
- CI jobs blocking PRs that violate layer boundaries
- Error messages with remediation guidance, not generic failures

"By enforcing invariants, not micromanaging implementations, they let agents ship fast without undermining the foundation."

**3. Garbage Collection / Entropy Management**
- Background Codex tasks scan for deviations regularly
- Auto-generate targeted refactoring PRs (reviewable in under 1 minute)
- Track quality grades per domain and architectural layer
- "Golden principles" encoded in repo
- Initially 20% of Friday time cleaning "AI slop" — replaced by continuous background processes

### Documentation as Code Infrastructure

```
docs/
  design-docs/     — architectural decisions
  exec-plans/      — active/completed execution plans, tech debt tracking
  product-specs/   — feature specs with navigation indices
  references/      — design system references formatted for LLMs
```

All versioned in repo. CI validation ensuring consistency.

### The 30/70 Rule

Codex handles ~70% of coding. Engineers handle ~30% (planning, architecture, judgment).

### Tiered Code Review

- AI code review: ~90% signal rate (9/10 comments valid, equal/better than human)
- Non-critical code: merge post-AI review without human sign-off
- Critical code (core agent, open source): mandatory human review

## Source 2: Shell + Skills + Compaction — Tips for Long-Running Agents

URL: https://developers.openai.com/blog/skills-shell-tips/

### Three Agentic Primitives

**Skills**: Versioned instruction bundles with SKILL.md manifest. Platform exposes name, description, path. Aligned with Agent Skills open standard at agentskills.io.

**Shell**: Terminal environment. Agents install deps, run scripts, write outputs within stateful request contexts.

**Compaction**: Manages context limits by automatically compressing history. Responses API offers both automatic in-stream compaction and standalone `/responses/compact` endpoint.

### 10 Key Recommendations

1. Design skill descriptions as routing logic, not marketing copy. Include "when to use" vs "when not to use"
2. Include negative examples and edge cases — Glean saw ~20% drop without them, recovered after adding
3. Embed templates within skills to avoid inflating unrelated queries
4. Plan for long runs through container reuse and compaction, not emergency fallback. Use `previous_response_id` for continuity
5. Use explicit skill invocation ("Use the X skill") when determinism matters
6. Treat skills with networking as high-risk: strict allowlists, confirmation controls
7. Use `/mnt/data` as standard artifact handoff boundary
8. Networking is two-layer: org-level allowlists define max, request-level must be subset
9. Use `domain_secrets` for authenticated API calls to prevent credential exposure
10. Use consistent APIs across cloud and local for flexible development

### Build Patterns

- **Pattern A** (simplest): Install deps → fetch data → write artifact
- **Pattern B** (repeatable): Encode workflow in skills, mount in shell, execute deterministically
- **Pattern C** (advanced): Skills as enterprise workflow carriers. Glean achieved 73% → 85% accuracy on Salesforce tasks

## Source 3: Unrolling the Codex Agent Loop

URL: https://openai.com/index/unrolling-the-codex-agent-loop/

### Agent Loop Architecture

Each turn: Assemble inputs → Run inference → Execute tools if invoked → Return results → Repeat until no tool calls → End with assistant message.

### The Quadratic Scaling Problem

Agent loop is quadratic in JSON sent to Responses API — each request includes full conversation history. Every subsequent request grows.

### Why Codex Avoids previous_response_id

Maintains stateless requests for Zero Data Retention compliance. Storing data for it would contradict ZDR.

### Prompt Caching Strategy

Cache hits require exact prefix matches. Structure: static content (instructions, examples) at beginning, variable content at end.

Cache miss causes: changing tools mid-conversation, changing model, changing sandbox config/approval/CWD. MCP tools are challenging — servers dynamically change tool lists.

Fix: Append new messages rather than modifying earlier ones.

### Compaction

Compact once token count exceeds threshold. `/responses/compact` returns items including special `type=compaction` item with `encrypted_content` preserving latent understanding.

## Source 4: Codex App Server Architecture

URL: https://openai.com/index/unlocking-the-codex-harness/

### Three Conversation Primitives

- **Item**: Atomic unit with lifecycle (started, delta, completed). Represents messages, tool executions, approval requests, diffs.
- **Turn**: Groups items from single unit of agent work.
- **Thread**: Durable container supporting creation, resumption, forking, archival.

### Communication

JSON-RPC streamed as JSONL over stdio. Server sends approval requests to client and pauses until allow/deny.

### MCP Comparison

OpenAI found maintaining MCP semantics difficult for richer sessions (streaming diffs, approval flows, thread persistence). Still supports Codex as MCP server for simpler workflows.

## Source 5: A Practical Guide to Building Agents (PDF)

URL: https://cdn.openai.com/business-guides-and-resources/a-practical-guide-to-building-agents.pdf

### Three Core Components

**Model**: Start with most capable, swap smaller to test. Not every task needs the smartest model.

**Tools** (three categories):
- Data tools: retrieve context (DB, files, web search)
- Action tools: send emails, update CRM, execute transactions
- Orchestration tools: agent-to-agent calls

Tool count isn't the issue — similarity/overlap is. Some succeed with 15+ distinct tools; others struggle with <10 overlapping ones.

**Instructions**: Break tasks into discrete steps. Define expected outputs clearly. Anticipate edge cases. "Vague instructions = unpredictable agent."

### Orchestration

Maximize single agent first. Multi-agent only when: tool overlap causes selection failures, complex branching logic becomes unmaintainable, separate non-overlapping tasks exist.

**Manager pattern**: Central orchestrator delegates to specialists.
**Decentralized pattern**: Direct agent-to-agent handoffs.

### Seven Guardrail Types

1. Relevance classifier (off-topic)
2. Safety classifier (jailbreaks, injection)
3. PII filter
4. Moderation layer
5. Tool safeguards (risk-rated: low/medium/high)
6. Rules-based (blocklists, regex)
7. Output validation (brand alignment, pricing errors)

## Source 8: OpenDev — Building AI Coding Agents for the Terminal (arXiv)

URL: https://arxiv.org/html/2603.05344v1

### Behavioral Steering Over Long Horizons

- Event-driven system reminders at decision points (not just initial prompts)
- Doom-loop detection after repeated failed tool calls
- Conditional prompt composition from 50+ independent sections ordered by priority (1-100)
- Addresses "instruction fade-out" where behavior drifts in long conversations

### Adaptive Context Compaction (5 Stages)

1. Summarize old observations
2. Move observations to external file
3. Compress message history summaries
4. Truncate oldest messages
5. Force exit if unrecoverable

Triggered at 80-90% token usage.

### Key Constants

- Compaction threshold: 80-90% token usage
- Max iterations: 25
- Tool timeout: 30 seconds
- Working memory: 10-15 turns
- Doom-loop: 3 identical failures or 5 errors in 5 iterations

### Performance Finding

Degradation beyond ~40% context utilization on 168K windows. "Smart Zone" (0-40%: focused) vs "Dumb Zone" (40%+: hallucinations, looping).

## Harness Engineering: Implementation Levels

### Level 1 (Basic) — 1-2 hours
AGENTS.md, pre-commit hooks, test suite, clear directory structure

### Level 2 (Team) — 1-2 days
AGENTS.md with team conventions, CI-enforced constraints, shared prompt templates

### Level 3 (Production) — 1-2 weeks
Custom middleware, observability, scheduled entropy management, harness versioning/A/B testing

### Minimum Viable Harness Checklist

- Small AGENTS.md (<100 lines, ideal <60) pointing to deeper docs
- Reproducible dev environment with per-worktree isolation
- Mechanical invariants in CI
- Agent legibility hooks (structured logs, queryable traces)
- Clear evaluation gates agents can run
- Safety rails (least-privilege, controlled egress, audit logs)
