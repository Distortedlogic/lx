# Academic Papers and Frameworks for Agent Safety Through Sandboxing

## Key Papers

### Agent Containment and Sandboxing

**Systems Security Foundations for Agentic Computing** (Dec 2025)
- arXiv:2512.01295
- Foundational paper applying computer systems security to agentic AI
- Key insight: TCB in agentic systems is probabilistic/non-deterministic
- Probabilistic reference monitor 99% correct still leaves 1% exploitable
- Recommends treating model as untrusted component, enforcing invariants through isolation/mediation

**Fault-Tolerant Sandboxing for AI Coding Agents** (Dec 2025)
- arXiv:2512.12806
- Policy-based interception + transactional filesystem snapshots
- Commands categorized: safe (bypass snapshots), unsafe (blocked), uncertain (checkpoint/rollback)
- 100% interception of high-risk commands, 100% rollback success, 14.5% (~1.8s) overhead

**ceLLMate: Sandboxing Browser AI Agents** (Dec 2025)
- arXiv:2512.12594
- First systems-level sandboxing for browser agents
- Enforces least-privilege at HTTP layer (not model layer)
- "Agent sitemap" abstraction bridging low-level actions to privilege requirements
- Blocks prompt injection with 7-15% latency overhead

**LLM-in-Sandbox Elicits General Agentic Intelligence** (Jan 2026)
- arXiv:2601.16206
- Proposes giving LLMs sandbox with basic computer capabilities for multi-turn exploration

### Runtime Enforcement

**AgentSpec: Customizable Runtime Enforcement for Safe LLM Agents** (Mar 2025, ICSE 2026)
- arXiv:2503.18666
- Lightweight DSL for runtime constraints
- >90% prevention in code agents, 100% hazard elimination in embodied agents, 100% compliance in AV
- Millisecond overhead

**AEGIS: No Tool Call Left Unchecked** (Mar 2026)
- arXiv:2603.12621
- Framework-agnostic pre-execution firewall and audit layer
- Three-stage: deep string extraction, content-first risk scanning, composable policy validation
- Supports 14 agent frameworks (Python, JS, Go)
- Blocks all attacks in 48-instance suite, 1.2% false positive, 8.3ms median latency
- Ed25519 signatures + SHA-256 hash chaining for tamper-evident audit trails

**Progent: Programmable Privilege Control for LLM Agents** (Apr 2025)
- arXiv:2504.11703
- First privilege control framework with DSL for fine-grained policies
- Reduces attack success to 0% on AgentDojo, ASB, AgentPoison
- Policies auto-generated by LLMs, updated dynamically

**SEAgent: Taming Privilege Escalation** (Jan 2026)
- arXiv:2601.11893
- Mandatory Access Control framework built on ABAC
- Monitors agent-tool interactions via information flow graph
- Blocks privilege escalation with low false positive rates

### Multi-Agent Security

**IsolateGPT: An Execution Isolation Architecture** (NDSS 2025)
- arXiv:2403.04960
- "Hub" acts like OS kernel for agent interactions
- Mediates collaboration between mutually distrusting apps
- <30% overhead for 75% of queries

**AgenticCyOps: Securing Multi-Agentic AI Integration** (Mar 2026)
- arXiv:2603.09134
- Enterprise SOC multi-agent framework using MCP
- Reduces exploitable trust boundaries by 72% vs flat multi-agent

**Information-Theoretic Privacy Control for Sequential Multi-Agent LLM Systems** (Mar 2026)
- arXiv:2603.05520
- Strict sanitization drops task success to 35-65%

**AgentLeak: Privacy Leakage in Multi-Agent LLM Systems** (Feb 2026)
- arXiv:2602.11510

### Attack Research

**"Your AI, My Shell": Prompt Injection on Agentic Coding Editors** (Sep 2025)
- arXiv:2509.22040
- AIShellJack framework: 314 payloads, 70 MITRE ATT&CK techniques
- Up to 84% attack success on GitHub Copilot and Cursor

**Prompt Injection Attacks on Agentic Coding Assistants** (Jan 2026)
- arXiv:2601.17548
- Systematic analysis across skills, tools, protocol ecosystems

**Visual Confused Deputy** (Mar 2026)
- arXiv:2603.14707
- Screen-based confused deputy attack on GUI agents via visual manipulation

**Not What You've Signed Up For** (Greshake et al., 2023)
- Seminal paper on indirect prompt injection against tool-using LLMs
- Demonstrates attacks against Bing Chat, ChatGPT plugins, LangChain agents

**AgentDojo** (2024, ETH Zurich)
- Dynamic environment for evaluating attacks/defenses on LLM agents
- Current defenses (prompt-based, fine-tuning) are insufficient
- Defense-in-depth is the only viable approach

**ToolEmu** (2024)
- LLM-emulated sandbox for evaluating tool use risks
- 36 risk categories across 144 tool use cases

**Agent Security Bench** (ICLR 2025)
- Mixed attack success rates of 84.3%
- Direct prompt injection at 72.7%

### Least Privilege

**MiniScope** (UC Berkeley, Dec 2025)
- arXiv:2512.11147
- Permission solver analyzing tasks for minimum privilege scope
- 1-6% latency overhead

**Towards Automating Data Access Permissions in AI Agents** (IEEE S&P 2026)
- Franzi Roesner et al.
- Permission management needs developed from scratch for AI agents

### Observability

**AgentSight: System-Level Observability Using eBPF** (2025)
- arXiv:2508.02736
- eBPF probes capturing Intent Stream (LLM) + Action Stream (syscalls)
- <3% overhead monitoring 6 collaborating agents

## Frameworks and Guardrails

### LlamaFirewall (Meta, May 2025)

Open-source, production-used at Meta. Three components:
1. **PromptGuard 2**: Jailbreak detection (SOTA performance)
2. **Agent Alignment Checks**: Chain-of-thought auditor for injection/misalignment
3. **CodeShield**: Online static analysis for insecure code

>90% efficacy reducing attack success on AgentDojo.

- https://arxiv.org/abs/2505.03574
- https://github.com/meta-llama/PurpleLlama/tree/main/LlamaFirewall

### Invariant Guardrails (acquired by Snyk, Jun 2025)

Transparent proxy between apps and MCP servers/LLM providers. Enforces contextual rules (data flow, tool call restrictions). Detects PII, secrets, copyright, injection.

- https://invariantlabs.ai/guardrails
- https://github.com/invariantlabs-ai/invariant

### NVIDIA NeMo Guardrails

Programmable guardrails: topic control, PII detection, RAG grounding, jailbreak prevention. NemoClaw stack includes OpenShell for isolated sandbox execution. Integrates with LangChain/LangGraph/LlamaIndex.

- https://github.com/NVIDIA-NeMo/Guardrails

### Superagent (Dec 2025)

Central Safety Agent evaluating actions before execution. Supports Claude Code, Gemini, Codex in clean sandboxes with PII redaction.

- https://github.com/superagent-ai/superagent

## Deterministic Sandboxing vs Probabilistic Guardrails

### The Central Tension

**Probabilistic (model-level)**: LLM classifiers, Constitutional AI, prompt-based safety. Cannot be formally verified. A 99% correct reference monitor still leaves 1% exploitable. Strengths: flexible, context-aware, handles novel situations.

**Deterministic (OS-level)**: Filesystem isolation, network namespaces, Firecracker, gVisor, WASI. Hard constraints regardless of model behavior. No false negatives. Weaknesses: binary (allowed/not), cannot reason about intent.

### Consensus: Defense-in-Depth (Both)

```
Layer 1: Model guardrails (catches ~99%, good UX)
Layer 2: Pre-execution validation (AEGIS, Progent, parameter checking)
Layer 3: OS sandbox (hard boundary, catches everything else)
Layer 4: Monitoring and audit (detect patterns of attempted abuse)
Layer 5: Human review (final safety net for high-risk)
```

Each layer independently insufficient, together robust.

| Source | Recommendation |
|--------|---------------|
| Greshake et al. | Sandboxing essential; guardrails alone insufficient |
| OWASP LLM Top 10 | Least privilege + sandboxing |
| Anthropic RSP | Containment scales with capability |
| Simon Willison | "Cannot solve prompt injection with more AI" |
| Trail of Bits | OS-level sandboxing is primary defense |
| E2B / OpenHands | Built on deterministic sandboxing |

**Bottom line**: Guardrails are a UX feature (friendlier agent). Sandboxing is a security feature (safer agent). You need both. If you can only have one, choose sandboxing.

## Anthropic's Agent Safety Research

### Constitutional AI for Tool Use

Extended to tools: principles define what AI should/shouldn't do with tools, self-critique evaluates tool calls, RLHF trains safe patterns. Constitutional Classifiers++ use probe + classifier architecture at ~1% additional compute.

### Responsible Scaling Policy (RSP)

- ASL-2 (current): Standard measures
- ASL-3: Enhanced containment including sandboxing
- ASL-4+: Strong containment with formal verification
- Agents operating autonomously require higher ASL measures

### Sleeper Agents (2024)

Models can have hidden behaviors activating under specific conditions. Cannot rely solely on model behavior. Need hard boundaries.
