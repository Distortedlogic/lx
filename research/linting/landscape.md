# Linting Landscape Survey

A comprehensive technical survey of how programming languages implement linting,
covering architecture, parsing, rule systems, plugin models, fix capabilities,
and performance characteristics.

---

## 1. Python

### 1.1 Pylint

**Repository:** https://github.com/pylint-dev/pylint

**Parsing approach:** Pylint uses [astroid](https://github.com/pylint-dev/astroid), a
custom AST library that wraps and extends Python's built-in `ast` module. Astroid
rebuilds the tree generated by `_ast` by recursively walking down the AST and
constructing extended nodes with additional methods for type inference, scope
resolution, and tree navigation. This is not a separate parser -- it piggybacks on
CPython's parser, then enriches the resulting tree.

**Checker types:** Pylint has three distinct checker categories:

1. **AST checkers** -- operate on the astroid AST. The most common type. Implement
   `visit_<node_type>` and `leave_<node_type>` callbacks (visitor pattern). For
   example, `visit_functiondef` fires when entering a function definition node,
   `leave_functiondef` when exiting.

2. **Token checkers** -- operate on the list of tokens produced by Python's
   `tokenize` module. Useful for whitespace, indentation, and encoding checks
   that are awkward at the AST level.

3. **Raw checkers** -- operate on the raw file stream (plain text). Used for
   checks like line length, file encoding, and BOM detection.

**Rule definition:** Checkers inherit from `BaseChecker` and declare a `msgs`
dictionary mapping message codes to tuples of (display template, symbol,
description). Message IDs use category prefixes: C (convention), W (warning),
E (error), F (fatal), R (refactoring). Checkers call `self.add_message()` to
emit diagnostics.

**Configuration:** `.pylintrc` or `pyproject.toml` `[tool.pylint]` section.
Supports enable/disable by message code or symbol, confidence level filtering,
and per-checker options. Confidence levels: HIGH, CONTROL_FLOW, INFERENCE,
INFERENCE_FAILURE, UNDEFINED.

**Plugin/extension:** Plugins are Python modules that export a `register(linter)`
function. The function receives the `PyLinter` instance and calls
`linter.register_checker(YourChecker(linter))`. Plugins are loaded via
`--load-plugins` CLI flag or config.

**Auto-fix:** Limited. Pylint is primarily a diagnostic tool; it does not have a
built-in auto-fix system comparable to ruff or eslint.

**Performance:** Single-threaded by default. Supports `-j N` for parallel
execution via multiprocessing, with `get_map_data()` / `reduce_map_data()`
methods on checkers enabling data aggregation across worker processes. Generally
the slowest of the Python linters due to astroid's inference engine.

**Error reporting:** Text, JSON, parseable, colorized, and custom formatter support.

Source: [Pylint checker docs](https://pylint.pycqa.org/en/latest/development_guide/how_tos/custom_checkers.html),
[Astroid](https://github.com/pylint-dev/astroid)

### 1.2 Flake8

**Repository:** https://github.com/PyCQA/flake8

**Architecture:** Flake8 is a *meta-linter* that glues together three core tools:

- **pyflakes** -- logical error detection (undefined names, unused imports)
- **pycodestyle** -- PEP 8 style checking
- **mccabe** -- cyclomatic complexity measurement

Flake8 provides a unified CLI, configuration, and output format.

**Parsing approach:** Each sub-tool parses differently:
- pyflakes compiles the source to a Python AST via `ast.parse()` and walks it
- pycodestyle operates on raw lines of text with regex-based pattern matching
- mccabe uses the AST for complexity calculation

Flake8 itself does not parse; it delegates to its plugins.

**Plugin system:** Flake8's primary strength. Plugins register via Python
`entry_points` (`flake8.extension` and `flake8.formatting` groups). There are
two plugin types:

- **AST plugins** -- receive the AST and filename, return violations
- **Physical line / logical line plugins** -- receive individual lines

Flake8 carries an in-tree shim for pyflakes (`flake8.plugins.pyflakes`).
Pycodestyle checks are registered as a group with individual checks mapped like
`pycodestyle.<check-name> = pycodestyle:<check-name>`.

**Configuration:** `setup.cfg`, `tox.ini`, or `.flake8` (INI format). Supports
`select`, `ignore`, `per-file-ignores`, `max-line-length`, and plugin-specific
options.

**Auto-fix:** None built-in. Third-party tools like autopep8 and autoflake
provide fixes for some flake8-reported issues.

**Performance:** Faster than pylint (no type inference), but slower than ruff
by 10-100x. Single-process, though plugins can be lightweight.

Source: [Flake8 docs](https://flake8.pycqa.org/en/latest/),
[Plugin handling](https://flake8.pycqa.org/en/7.0.0/internal/plugin_handling.html)

### 1.3 Ruff

**Repository:** https://github.com/astral-sh/ruff

**Parsing approach:** Ruff uses a **hand-written recursive descent parser**
implemented in Rust (`ruff_python_parser` crate). Evolution:

1. Initially borrowed RustPython's LALRPOP-generated parser
2. Maintained a fork in the ruff repo
3. Switched to a hand-written parser in v0.4.0 (by Victor Hugo Gomes)

The hand-written parser is 2x faster than the LALRPOP version, translating to
20-40% overall speedup. It uses **Pratt parsing** for expression precedence.
Critically, the parser has **error recovery** -- it can parse invalid Python,
which is essential for a linter (users lint broken code).

The parser produces an AST (not CST). AST nodes are defined in `ruff_python_ast`:
- `Mod` -- module containers
- `Stmt` -- 50+ statement variants
- `Expr` -- 45+ expression variants
- `Pattern` -- match patterns (PEP 634)

All nodes carry `range: TextRange` for source location.

**Architecture (50+ crate workspace):**

| Layer | Crates | Purpose |
|-------|--------|---------|
| Entry points | `ruff` CLI, `ruff_server` LSP, `ruff_wasm` | User-facing binaries |
| Analysis engines | `ruff_linter`, `ruff_python_formatter`, `ty_python_semantic` | Core logic |
| Language infra | `ruff_python_parser`, `ruff_python_ast`, `ruff_python_semantic` | Parsing, AST, name binding |
| Infrastructure | `ruff_workspace`, `ruff_db`, `ruff_cache` | Config, Salsa, caching |

**Checker pipeline:**
1. Parse source to AST
2. Build semantic model via `SemanticModelBuilder` (scopes, bindings, use-def chains)
3. Run `Checker::check()` which traverses AST nodes
4. Apply rules (pattern matching on `StmtKind`, `ExprKind`, etc.)
5. Generate diagnostics with optional fixes

**Rule system:** 800+ rules reimplemented natively in Rust. Rules are organized
into ~70 categories mirroring their source tools:
- Pyflakes (F), Pycodestyle (E, W), Pylint (PL)
- flake8-bugbear (B), isort (I), pyupgrade (UP)
- Ruff-specific (RUF), plus 60+ more families

Selection uses prefix matching: exact codes (`E501`), categories (`E`),
sub-categories (`E70`), wildcards (`PL*`).

**Configuration:** `ruff.toml`, `.ruff.toml`, or `pyproject.toml` `[tool.ruff]`.
Three-level hierarchy: CLI args > project config > user config (`~/.config/ruff/ruff.toml`).
`LinterSettings` has 100+ options.

**Fix system:** Fixes categorized by safety:
- **Safe** -- guaranteed semantic preservation (e.g., F401 unused imports)
- **Unsafe** -- may alter behavior (e.g., B006 mutable defaults)
- **Display** -- shown but not auto-applied

Unsafe fixes require `--unsafe-fixes` flag. Supports iterative application until
convergence with conflict detection.

**Suppression:** `# noqa: F841`, file-level `# ruff: noqa`, block-level
`# ruff: disable[N803]` ... `# ruff: enable[N803]`.

**Performance:** 10-100x faster than Python-based linters. CPython codebase:
pylint ~2.5 min, ruff ~0.3s. Key factors:
- Rust: zero-cost abstractions, no GC
- Parallel file processing via **rayon** (`par_iter()`)
- Per-package caching (SHA-256 of content + config + version)
- Single AST pass for all rules
- Error-recovery parser avoids re-parsing

Source: [Ruff blog on v0.4.0 parser](https://astral.sh/blog/ruff-v0.4.0),
[Ruff internals](https://compileralchemy.substack.com/p/ruff-internals-of-a-rust-backed-python),
[DeepWiki Ruff overview](https://deepwiki.com/astral-sh/ruff/1-ruff-overview)

### 1.4 Pyflakes

**Repository:** https://github.com/PyCQA/pyflakes

Standalone tool for detecting logical errors. Uses `ast.parse()` from the
standard library. Walks the AST with a visitor, tracking scope and name bindings.
No style checking, no configuration file, no plugin system. Extremely fast due
to simplicity. Primarily consumed through flake8.

---

### 1.5 Pycodestyle (formerly pep8)

**Repository:** https://github.com/PyCQA/pycodestyle

PEP 8 style checker. Does **not** use an AST -- operates on raw lines of text
with regex matching and simple state machines. Each check is a function that
receives either a physical line or a logical line (tokens joined). Very fast
but limited to surface-level style checks. No type awareness.

## 2. Rust: Clippy

**Repository:** https://github.com/rust-lang/rust-clippy

**Compiler integration:** Clippy is **deeply integrated with rustc**. The
`clippy-driver` binary wraps `rustc_driver` and implements the `Callbacks` trait
to hook into the compilation pipeline. It does not have its own parser or AST --
it operates directly on rustc's internal representations.

**Architecture (6 crates):**

| Crate | Purpose |
|-------|---------|
| `clippy` (root) | Produces `cargo-clippy` and `clippy-driver` binaries |
| `clippy_lints` | ~800 lint implementations, exports `register_lint_passes` |
| `clippy_utils` | Shared analysis utilities for HIR/type inspection |
| `clippy_config` | `Conf` struct, parses `clippy.toml` |
| `declare_clippy_lint` | `declare_clippy_lint!` macro and metadata |
| `clippy_dev` | Developer tooling (`cargo dev` subcommands) |

**Lint pass system:** Two fundamental pass types:

**EarlyLintPass:**
- Runs on the **AST** after macro expansion, before HIR lowering
- No access to type information or symbol resolution
- Faster execution
- Context: `EarlyContext`
- Use case: syntax-only lints (naming conventions, attribute checks)

**LateLintPass:**
- Runs on the **HIR** (High-level Intermediate Representation) after type checking
- Full access to type info via `TyCtxt<'tcx>`, `typeck_results()`, trait resolution
- Context: `LateContext`
- Most Clippy lints (~95%) use this pass because they need type information

**Lint declaration:** The `declare_clippy_lint!` macro generates:
- A static `Lint` reference matching rustc's expectations
- A `LINT_NAME_INFO` constant of type `&LintInfo`
- Metadata for docs and group registration

```rust
declare_clippy_lint! {
  /// Documentation shown by --explain
  #[clippy::version = "1.XX.0"]
  pub LINT_NAME,
  category,
  "short description"
}
```

**Lint registration:** `register_lint_passes()` receives `&mut LintStore` and
`&'static Conf`, then:
1. Registers renamed/removed lints
2. Pushes early pass constructors via `store.early_passes.extend(...)`
3. Pushes late pass constructors via `store.late_passes.extend(...)`
4. Groups lints by category via `LintListBuilder`

**Categories:**

| Category | Default | Purpose |
|----------|---------|---------|
| correctness | deny | Outright wrong or useless code |
| suspicious | warn | Most likely wrong or useless |
| style | warn | Should be written more idiomatically |
| complexity | warn | Simple thing done in complex way |
| perf | warn | Can be written faster |
| pedantic | allow | Stricter, occasional false positives |
| restriction | allow | Prevent specific language features |
| nursery | allow | New, under development |
| cargo | allow | Cargo.toml issues |

`clippy::all` = correctness + suspicious + style + complexity + perf.

**Configuration:** `clippy.toml` or `.clippy.toml`. Parsed by serde into a `Conf`
struct with fields for MSRV, thresholds (cognitive complexity, array size),
allowlists (disallowed methods/names), and behavior flags. TOML keys use
kebab-case.

**Utilities (`clippy_utils`):**
- `diagnostics`: `span_lint_and_sugg()` for fix suggestions
- `sugg`: `Sugg` builder for constructing code suggestions
- `consts`: `ConstEvalCtxt` for compile-time evaluation
- `ty`: Type inspection helpers
- `hir_utils`: `SpanlessEq` for structural comparison
- `visitors`: `for_each_expr()`, `ExprUseVisitor`

**Auto-fix:** Clippy emits `MachineApplicable` suggestions via rustc's suggestion
system. Applied with `cargo clippy --fix`. Applicability levels: MachineApplicable,
MaybeIncorrect, HasPlaceholders, Unspecified.

**MSRV handling:** Lints can conditionally enable suggestions based on the
configured minimum supported Rust version. Set via `clippy.toml`, `Cargo.toml`
`rust-version`, or `#[clippy::msrv = "1.XX.0"]` attributes.

**Plugin/extension:** Clippy does not support external plugins. All lints are
first-party and ship with the Rust toolchain.

Source: [Clippy lint passes](https://doc.rust-lang.org/nightly/clippy/development/lint_passes.html),
[DeepWiki Clippy](https://deepwiki.com/rust-lang/rust-clippy),
[Clippy adding lints](https://doc.rust-lang.org/nightly/clippy/development/adding_lints.html)

## 3. JavaScript / TypeScript

### 3.1 ESLint

**Repository:** https://github.com/eslint/eslint

**Parsing approach:** ESLint uses **Espree** as its default parser. Espree is
a fork of Esprima, rebased on **Acorn** (a small, fast JS parser). It produces
an **ESTree**-compliant AST. ESTree is the standard AST format for JavaScript
tooling.

For TypeScript: `@typescript-eslint/parser` converts TypeScript into
ESTree-compatible form. For JSX/Flow: `@babel/eslint-parser` uses Babel's
parser. Custom parsers implement `parseForESLint()` returning `{ ast, services,
scopeManager, visitorKeys }`.

**Architecture layers:**

1. **CLI** (`bin/eslint.js` -> `lib/cli.js`): argument parsing, file I/O, formatters
2. **CLIEngine**: reads config files (`.eslintrc`, `eslint.config.js`), manages
   plugin/parser loading, coordinates execution
3. **Linter** (`lib/linter/`): core engine. Parses text, retrieves AST, runs rules.
   No file I/O, no console output.
4. **SourceCode**: parsed source representation with token/comment access
5. **Rules** (`lib/rules/`): individual rule implementations

**AST traversal:** Uses `estraverse` for bidirectional tree walking. At each
node, the Linter emits an event matching the node type (e.g., `Identifier`).
On ascent, emits `Identifier:exit`. Rules register for these events.

**Rule definition:** Rules export `{ meta, create }`:

```javascript
module.exports = {
  meta: {
    type: "problem" | "suggestion" | "layout",
    docs: { description, url },
    messages: { msgId: "template with {{ placeholder }}" },
    fixable: "code" | "whitespace",
    hasSuggestions: true,
    schema: [/* JSON Schema for options */],
    defaultOptions: [/* defaults */]
  },
  create(context) {
    return {
      Identifier(node) { /* inspect node */ },
      "FunctionExpression:exit"(node) { /* on exit */ }
    };
  }
};
```

Rules receive a `context` object providing:
- `context.report({ node, messageId, data, fix, suggest })` -- emit diagnostic
- `context.sourceCode` -- getText, getScope, getTokens, getAncestors, etc.
- `context.options` -- user-configured rule options

**AST selectors:** CSS-like syntax for matching node patterns. Rules can listen
on selectors like `CallExpression[callee.name='eval']` instead of manually
filtering in the visitor.

**Fix system:** Fixer methods: `insertTextBefore/After`, `remove`, `replaceText`,
`replaceTextRange`. Fixes return single operations or generators yielding
multiple non-overlapping operations. Suggestions are manual-apply alternatives.

**Plugin system:** Plugins are npm packages exporting configs, processors, and/or
rules. Flat config (`eslint.config.js`) uses an array of config objects evaluated
in order. Plugins declare rules that users enable in their config.

**Configuration:** Flat config (eslint.config.js/mjs/cjs) replaced legacy
`.eslintrc.*`. Config objects specify `files` globs, `plugins`, `rules`
(with severity 0/1/2 or off/warn/error), `languageOptions` (parser, ecmaVersion,
globals), and `settings`.

**Performance:** JavaScript-based, single-threaded. `--cache` stores results
per file (content hash + config hash). Cache is fundamentally limited by
cross-file analysis (typed linting invalidates everything). `TIMING=1` flag
profiles per-rule execution time.

Source: [ESLint architecture](https://eslint.org/docs/latest/contribute/architecture/),
[Custom rules](https://eslint.org/docs/latest/extend/custom-rules)

### 3.2 Biome

**Repository:** https://github.com/biomejs/biome

**Parsing approach:** Biome uses a **lossless Concrete Syntax Tree (CST)** based
on the **Red-Green tree pattern**, via an internal fork of **rowan** (the library
from rust-analyzer). This architecture originated in Microsoft's Roslyn C#
compiler and was adopted by Swift's libsyntax, then rust-analyzer.

**Red-Green tree:**
- **Green tree** (inner): position-independent, contains actual source text.
  Nodes store only a `SyntaxKind` (u16 enum) and textual width. Immutable and
  can be shared.
- **Red tree** (outer): constructed on-demand over the green tree. Adds parent
  pointers and absolute offsets. This is what rule authors interact with.

The CST preserves **all** source information including whitespace and comments
("trivia"). Leading trivia precedes tokens (including line breaks); trailing
trivia follows tokens until the next line break.

**Error recovery:** The parser is both resilient (resumes after errors) and
recoverable (produces correct structure around errors). "Bogus nodes" decorate
broken syntax sections, protecting rule consumers from invalid trees.

**Architecture:** Biome is a unified toolchain (linter + formatter + future type
checker) in a single Rust binary. Components:

- **Scanner:** crawls filesystem, discovers configs, builds module graph
- **Parser:** per-language parsers (JS, TS, JSX, CSS, JSON, GraphQL, HTML)
  producing language-specific CSTs
- **Analyzer:** rule engine that traverses CST nodes
- **Formatter:** Prettier-compatible, operates on same CST
- **Daemon:** long-running server for editor integration (LSP)

**Rule engine (Rule trait):**

Rules implement the `Rule` trait with associated types:

| Type | Purpose |
|------|---------|
| `Query` | Which CST nodes trigger the rule (`Ast<JsDebuggerStatement>`, `Semantic<JsCallExpression>`) |
| `State` | Data flowing from `run()` to `diagnostic()`/`action()` |
| `Options` | User configuration from `biome.json` |
| `Signals` | Return type from `run()` |

Methods:
- `run(ctx)` -- called per matching node, returns `Option<State>` to trigger
- `diagnostic(ctx, state)` -- constructs rich diagnostic messages
- `action(ctx, state)` -- optional code fix with applicability level

**Rule declaration:**
```rust
declare_lint_rule! {
    pub NoDebugger {
        version: "1.0.0",
        name: "noDebugger",
        language: "js",
        recommended: true,
        sources: &[RuleSource::Eslint("no-debugger")],
        severity: Severity::Error,
    }
}
```

File-system based discovery: placing a rule in `src/lint/<group>/<name>.rs`
automatically registers it.

**Query types:** AST queries examine syntax only. Semantic queries access scope,
binding, and (future) type information.

**Rule categories:** a11y, complexity, correctness, nursery, performance,
security, style, suspicious. 450+ rules total.

**Configuration:** `biome.json` or `biome.jsonc`. Rule options generate
`configuration_schema.json` for IDE support.

**Fix system:** Actions provide automated corrections marked as `Always`
(guaranteed safe) or `MaybeIncorrect` (needs user verification).

**Performance:** Parses each file once, runs formatting and linting in the same
pass. 15x faster than ESLint on benchmarks (~200ms vs 3-5s for 10k-line
monorepo). Rust concurrency and multi-threading via rayon.

**Type-aware linting:** Biome is building its own type synthesizer rather than
depending on the TypeScript compiler. This is a work-in-progress.

Source: [Biome architecture](https://biomejs.dev/internals/architecture/),
[Rule engine](https://deepwiki.com/biomejs/biome/5.2-rule-engine-architecture),
[Biome linter](https://biomejs.dev/linter/)

## 4. Go

### 4.1 go vet

**Part of:** Go standard toolchain

**Architecture:** `go vet` is an analysis **driver** built on the
`golang.org/x/tools/go/analysis` framework. It loads and runs a set of built-in
analyzers (printf, shadow, structtag, etc.).

**Analysis framework core type:**

```go
type Analyzer struct {
    Name             string
    Doc              string
    Flags            flag.FlagSet
    Run              func(*Pass) (interface{}, error)
    RunDespiteErrors bool
    ResultType       reflect.Type
    Requires         []*Analyzer
    FactTypes        []Fact
}
```

**Pass object:** Provided to `Run`, contains:
- `Fset`, `Files` (parsed Go files), `Pkg`, `TypesInfo` (type-checked data)
- `ResultOf` -- results from required analyzers
- `Report(Diagnostic)` -- emit violations
- `ExportObjectFact` / `ImportObjectFact` -- cross-package facts

**Dependency system:** Analyzers declare dependencies via `Requires`. The driver
topologically sorts and runs them. Results flow through `Pass.ResultOf`.
Dependencies are acyclic (validated by `Validate()`).

**Facts (cross-package analysis):** The key innovation. Facts are serializable
data (`encoding/gob`) that an analyzer can export about objects in the current
package. When analyzing a dependent package, those facts become available via
`ImportObjectFact`. This enables **modular analysis**: the printf checker can
discover that `log.Fatalf` wraps `fmt.Printf`, record this as a fact, and then
check calls to `log.Fatalf` in other packages.

**Built-in analyzers providing infrastructure:**
- `inspect` -- efficient AST traversal for downstream analyzers
- `ctrlflow` -- control-flow graphs per function
- `buildssa` -- SSA-form intermediate representation

**Diagnostics:**
```go
type Diagnostic struct {
    Pos, End       token.Pos
    Category       string
    Message        string
    URL            string
    SuggestedFixes []SuggestedFix
    Related        []RelatedInformation
}
```

**Driver programs:** `singlechecker` (one analyzer), `multichecker` (multiple),
`unitchecker` (build-integrated). The framework is designed for use in IDEs,
CI, code review tools, and batch pipelines.

Source: [go/analysis docs](https://pkg.go.dev/golang.org/x/tools/go/analysis)

### 4.2 Staticcheck

**Repository:** https://github.com/dominikh/go-tools

**Architecture:** Built on `go/analysis` framework, but adds **SSA-based
analysis** for deeper checks. Uses `golang.org/x/tools/go/ssa` to construct
Static Single Assignment form, enabling data-flow analysis, interprocedural
checks, and nil dereference detection.

**Check categories:**

| Prefix | Name | Focus |
|--------|------|-------|
| SA | Staticcheck | Correctness: bugs, stdlib misuse, concurrency |
| S | Simple | Code simplification |
| ST | Stylecheck | Naming, documentation, formatting conventions |
| QF | Quickfix | Automated refactorings (used by gopls) |
| U | Unused | Dead code detection |

150+ checks total. SA is the largest category.

**Configuration:** `staticcheck.toml`. Supports check selection/exclusion,
per-package configuration, and Go version targeting.

Source: [Staticcheck docs](https://staticcheck.dev/docs/checks),
[go-tools repo](https://github.com/dominikh/go-tools)

### 4.3 golangci-lint

**Repository:** https://github.com/golangci/golangci-lint

**Architecture:** A **meta-linter** that aggregates 50+ individual linters
(including go vet, staticcheck, revive, gocritic, etc.) into a single tool.

**Key features:**
- Parallel execution of independent linters
- Intelligent caching (per-package, content-hash based)
- Unified output format
- Single `.golangci.yml` configuration file
- New-issue-only mode (show only issues in changed code)

## 5. Other Notable Linters

### 5.1 ShellCheck (Shell)

**Repository:** https://github.com/koalaman/shellcheck

**Implementation language:** Haskell

**Parsing:** Custom parser built with **Parsec** (Haskell parser combinator
library). Produces an AST and zero or more parser-level warnings. The parser
handles Bash, POSIX sh, Dash, and ksh syntax.

**Parser design lessons (from the author):**
- Initially character-based per-construct (`${..}`, `$(..)`, `$'..'`), causing
  excessive backtracking on `$` symbols. A lookahead grouping approach improved
  performance by 10%.
- Column counting used Parsec's default tab stop of 8, creating alignment bugs.
  Recommendation: use tab stop of 1 (like GCC).
- Original design captured only start positions. Full token spans are necessary
  for diagnostics and auto-fix.

**Rule system:** Rules are `CommandCheck` objects with metadata including
shell-specific constraints (Sh, Bash, Dash). Checks are filtered by shell
dialect before execution, avoiding unnecessary invocations.

**Warning IDs:** Each warning has a unique SC-number (e.g., SC2162) with
dedicated documentation pages containing examples, fixes, and educational
rationale.

**Testing:** ~1500 unit tests via QuickCheck. Tests are colocated with
implementations using simple `verify`/`verifyNot` boolean predicates.

**Auto-fix:** Supported via text edits derived from token spans (feature added
after capturing full spans).

**Performance notes:**
- Multithreading was dropped due to unresolved Haskell space leaks
- Converting to ReaderT monad caused 10% regression
- Runtime dependencies exceed 100MB despite 8MB standalone executable

**Output formats:** JSON, XML, human-readable, GCC-compatible, TTY.

**Suppression:** `# shellcheck disable=SC2162` comments. Per-line suppression
recommended over per-statement.

Source: [ShellCheck lessons blog](https://www.vidarholen.net/contents/blog/?p=859),
[ShellCheck repo](https://github.com/koalaman/shellcheck)

### 5.2 RuboCop (Ruby)

**Repository:** https://github.com/rubocop/rubocop

**Parsing:** Uses the **parser** gem to produce an AST from Ruby source code.
The `rubocop-ast` library wraps this with additional methods. Developers can
inspect AST structure with `ruby-parse` CLI utility.

**Rule system (Cops):** Each rule is called a "cop" and inherits from
`RuboCop::Cop::Base`. Cops use the visitor pattern with callbacks:
- `on_send` -- method calls (the most common)
- `on_def` -- method definitions
- `on_class` -- class definitions
- `on_block` -- block constructs

Unlike classical visitors, cops only implement callbacks for nodes they care
about; non-existent methods are skipped during traversal.

**NodePattern (AST regex):** RuboCop's killer feature. A DSL for matching AST
structures, analogous to regex for strings:

- `def_node_matcher` -- checks the current node against a pattern
- `def_node_search` -- searches descendants for matches

Patterns support predicates (`nil?`), captures (`$_`), and ellipsis (`...`).
Method names ending in `?` return booleans.

**RESTRICT_ON_SEND:** Performance optimization. Cops declare a constant listing
the method names they care about, so `on_send` is only called for matching
method names instead of every send node.

**Departments:** Cops are organized into departments:
- **Layout** -- indentation, spacing, line length
- **Lint** -- potential errors, suspicious constructs
- **Style** -- Ruby style guide conventions
- **Naming** -- naming conventions
- **Security** -- security-related checks
- **Metrics** -- complexity measurements

**Auto-correction:** Cops extend `AutoCorrector` and provide a block to
`add_offense` receiving a `corrector` object. The corrector uses
`Parser::Source::TreeRewriter` for source transformations: insert, replace,
remove, wrap.

**Configuration:** `.rubocop.yml` (YAML). Per-cop enable/disable, severity
levels, custom parameters, file inclusion/exclusion patterns.
`ConfigurableEnforcedStyle` mixin for cops with multiple valid styles.

**Plugin system:** Gems created with `rubocop-extension-generator`. Extensions
register default configs via an `Inject` module. Major extensions: rubocop-rails,
rubocop-rspec, rubocop-performance.

**Testing:** `expect_offense` / `expect_no_offenses` / `expect_correction`
helpers with inline offense markers showing expected violation locations.

Source: [RuboCop custom cops](https://evilmartians.com/chronicles/custom-cops-for-rubocop-an-emergency-service-for-your-codebase),
[NodePattern docs](https://docs.rubocop.org/rubocop-ast/node_pattern.html),
[RuboCop development guide](https://docs.gitlab.com/development/rubocop_development_guide/)

### 5.3 Credo (Elixir)

**Repository:** https://github.com/rrrene/credo

**Architecture:** Written in Elixir. Focuses on code consistency and teaching
rather than correctness. Uses Elixir's built-in `Code.string_to_quoted/2` for
parsing (returns Elixir AST).

**Check categories:**
- **Consistency** -- coding style consistency (tabs vs spaces)
- **Readability** -- how easy code is to digest
- **Refactoring** -- opportunities to avoid technical debt
- **Warning** -- potentially dangerous patterns (e.g., leftover `IEx.pry`)
- **Design** -- code design issues

**Plugin system:** Plugins are modules providing an `init/1` callback. Credo's
execution consists of several pipeline steps; plugins can prepend/append tasks,
register configs, register commands, and register CLI switches.

**Configuration:** `.credo.exs` (Elixir script). Supports check selection,
priority levels, and file inclusion/exclusion.

Source: [Credo repo](https://github.com/rrrene/credo),
[Credo config docs](https://hexdocs.pm/credo/config_file.html)

### 5.4 Luacheck (Lua)

**Repository:** https://github.com/lunarmodules/luacheck

**Implementation:** Written in Lua itself. Includes a custom parser producing
AST nodes in near-MetaLua format. Parser state tracks range info for tokens;
all AST nodes carry range info including parentheses around expressions.

**Capabilities:** Detects undefined globals, unused variables/values,
uninitialized variable access, unreachable code. Supports Lua 5.1-5.4 and
LuaJIT syntax.

**Configuration:** `.luacheckrc` (Lua script). Supports per-file settings,
global whitelisting, and standard library definitions.

Source: [Luacheck repo](https://github.com/lunarmodules/luacheck)
