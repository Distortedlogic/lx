## Python

### The LL(1) Era (Python 1.0 through 3.8)

Python originally used an LL(1) parser generated from a custom grammar file (`Grammar/Grammar`). The parser generator (`pgen`) produced parse tables that drove a table-driven LL(1) parser, producing a Concrete Syntax Tree (CST) that was then converted to an Abstract Syntax Tree (AST).

Limitations of the LL(1) approach:

- **No left recursion**: The natural expression rule `expr: expr '+' term | term` had to be rewritten as `expr: term ('+' term)*`, producing flat lists instead of left-recursive trees, requiring post-processing to build proper AST nodes.
- **Ambiguous first sets**: When two alternatives in a rule shared tokens in their first sets, the grammar had to be contorted with artificial intermediate rules.
- **Grammar-AST disconnect**: Many constraints were enforced only during the CST-to-AST conversion, not during parsing. The grammar accepted syntactically invalid programs, and the compiler had to reject them later.
- **Intermediate CST overhead**: The CST consumed significant memory with chains of single-child nodes that served no purpose beyond satisfying the parser's requirements.

### The PEG Era (Python 3.9+)

PEP 617 introduced a PEG (Parsing Expression Grammar) parser, generated by the `pegen` parser generator. The transition happened in phases:

- Python 3.9: Both parsers shipped; flags allowed switching between them.
- Python 3.10: The old LL(1) parser was removed.

### PEG Parser Architecture

The new parser is generated from `Grammar/python.gram` by the `pegen` tool (located in `Tools/peg_generator/`). The grammar file format supports:

```
rule_name[return_type]:
    | alternative1 action1 { action_code }
    | alternative2 { action_code }
```

Key grammar features:
- **Ordered choice** (`|`): Alternatives are tried in order; the first match wins. This eliminates ambiguity by construction.
- **Sequencing** (`e1 e2`): Match both expressions in order.
- **Optional** (`[e]` or `e?`): Zero or one occurrence.
- **Repetition** (`e*`, `e+`): Zero-or-more or one-or-more.
- **Separated repetition** (`s.e+`): One or more occurrences of `e` separated by `s`.
- **Lookahead** (`&e`, `!e`): Positive/negative assertions without consuming input.
- **Commit** (`~`): Once reached, forces the current alternative regardless of subsequent failure (prevents backtracking past this point).
- **Named sub-expressions** (`name=expr`): Bind a matched sub-expression to a name for use in actions.

Actions are Python or C expressions enclosed in `{ }` that construct AST nodes directly, eliminating the CST intermediate representation entirely.

### Left Recursion in PEG

PEG parsers traditionally cannot handle left recursion (infinite recursion on the same input position). CPython's pegen implements left recursion through strategic use of the memoization cache:

1. On first encounter of a left-recursive rule at position P, the memo cache returns a failure sentinel.
2. The rule falls through to non-left-recursive alternatives and produces a partial result.
3. The rule is re-invoked at position P, and the memo cache now returns the partial result.
4. The rule uses this partial result to produce a longer match.
5. Steps 3-4 repeat until no longer match is possible.

This technique supports simple left recursion, indirect left recursion (A -> B -> A), and hidden left recursion.

### Packrat Parsing and Memoization

The PEG parser uses packrat parsing (memoization of rule results at each input position) to guarantee O(n) parsing time despite unlimited lookahead. The trade-off is increased memory consumption. However, this is offset by eliminating the CST:

**Benchmark on 100,000 lines of arithmetic expressions:**
- New PEG parser: 1.28s, 681 MiB
- Old LL(1) parser: 1.44s, 836 MiB

**Benchmark on Python 3.8 stdlib (749,570 lines):**
- New PEG parser: 3.29s, 77 MiB
- Old LL(1) parser: 3.37s, 70 MiB

The PEG parser is faster in both cases, and uses less memory on large inputs due to CST elimination, though slightly more on typical code.

## Rust

### Architecture

The Rust compiler uses a hand-written recursive descent parser in `rustc_parse`. The parser is organized by semantic construct: `parse_item()`, `parse_stmt()`, `parse_expr()`, `parse_ty()`, `parse_pat()`, etc. Each function corresponds to a grammar production, and the overall structure mirrors the Rust grammar closely.

### Parser Organization

The parser lives in `rustc_parse::parser`, with entry points in the parent `rustc_parse` crate that transform source files or token streams into a `Crate` (the root AST node). The `Parser` struct holds:

- A token cursor over the token stream
- The current and previous token
- Expected-token tracking for error messages
- Restriction flags (e.g., "no struct literals in this context")
- The parent `ParseSess`, which contains the `SourceMap`, diagnostics infrastructure, and parsing metadata

Both the `Lexer` and `Parser` use lifetime parameters binding them to the `ParseSess`, minimizing data copying.

### Expression Parsing

Rust uses a precedence-climbing approach (equivalent to Pratt parsing) for expression parsing. The `parse_expr_res()` method takes restriction flags, and binary operator precedence is encoded in a lookup table. The parser handles Rust's complex expression grammar including:

- Block expressions (`{ ... }`)
- Closures (`|args| body`)
- `if`/`match`/`loop`/`while`/`for` as expressions
- `?` (try operator, postfix)
- Range operators (`..`, `..=`)
- Type ascription and cast (`as`)

### Error Recovery

Rust's parser is renowned for its error recovery quality. The general strategy is to **parse a superset of valid Rust**, accepting some invalid constructs during parsing and emitting errors alongside the AST. This allows the parser to continue and report multiple errors per compilation.

Specific techniques:
- **Recovery parsing**: When an expected token is missing, the parser may insert a synthetic token and continue. For example, a missing semicolon after a statement does not stop parsing.
- **Helpful suggestions**: The parser detects common mistakes and suggests corrections (e.g., `=` instead of `==` in a pattern, `fn` instead of `function`).
- **Unicode character suggestions**: When the parser encounters a visually similar Unicode character (e.g., fullwidth `=`), it suggests the ASCII equivalent.
- **Context tracking**: The parser tracks context (e.g., "we're inside an `if` condition") to provide contextual error messages.

A tension exists between error recovery and macro parsing: because the parser recovers by accepting a superset of valid Rust, macro fragment specifiers like `expr` and `stmt` may match more token streams than they should, creating backward-compatibility hazards.

### Macro Parsing

During initial parsing, macro definitions and invocations are not expanded. Macro invocation bodies are stored as token streams. Macro expansion happens in a later phase, which may recursively invoke the parser to parse the expanded tokens. This lazy approach avoids ordering issues (macros can reference items defined after them).

## JavaScript

### Acorn

Acorn is a small, fast JavaScript parser written in JavaScript. It produces an ESTree-compatible AST and is used as the basis for many JavaScript tools. Key characteristics:

- Hand-written recursive descent parser
- Full ES2024+ support
- Plugin system for extending the grammar
- Separate tokenizer accessible via `acorn.tokenizer()`
- Approximately 1,313 ops/sec parsing jquery.js

### Babel Parser

Babel's parser (`@babel/parser`, originally Babylon) was forked from Acorn but has diverged significantly. It adds:

- JSX support
- TypeScript support
- Flow support
- A powerful plugin system for experimental syntax proposals
- Preservation of comments and source positions for code transformation
- Approximately 1,250 ops/sec parsing jquery.js

Both Acorn and Babel use recursive descent with explicit precedence handling for expressions.

### V8: Eager Parsing and Preparsing

V8 implements a two-tier parsing system to minimize startup latency:

**Full Parser**: Produces a complete AST and compiles it to bytecode. Used for code that will execute immediately.

**Preparser**: A lightweight parser that performs minimal validation while skipping function bodies. It verifies syntactic validity and collects variable metadata but does not produce an AST or bytecode.

#### Decision Mechanism

By default, function declarations and expressions are preparsed. Eager compilation is triggered for "Possibly-Invoked Function Expressions" (PIFEs):
- Parenthesized function expressions: `(function(){...})`
- UglifyJS-style patterns: `!function(){...}()`, `,function(){...}()`

The preparser assumes these will be called immediately and compiles them eagerly to avoid the cost of preparsing followed by full parsing.

#### Variable Tracking

The preparser performs full scope analysis:
- Tracks variable declarations and references to determine stack vs. heap allocation
- Resolves shadowing and variable binding
- Serializes variable allocation as a dense array of flags per variable
- Stores metadata only for functions containing inner functions (most don't need it)

When a preparsed function is finally called:
1. Full parsing and compilation occur on-demand.
2. Inner functions are skipped entirely (not re-preparsed) because their variable metadata was serialized.
3. This guarantees each function is preparsed at most once and fully parsed at most once, eliminating the O(n^2) worst case of recursive preparsing in deeply nested code.

#### Shared Code Architecture

V8 originally had a standalone preparser implementation that diverged from the full parser. They unified both through a shared `ParserBase` template class using the Curiously Recurring Template Pattern (CRTP), ensuring both paths handle syntax identically.

#### Ambiguous Syntax

A key challenge is that the parser must decide whether to eagerly compile before examining a function body. For example, `({ d }` could be destructuring or an arrow function parameter—both syntactically valid in different contexts, requiring tentative parsing.

## Lua

### Single-Pass Architecture

Lua's parser (`lparser.c`) is a single-pass recursive descent parser that directly emits bytecode during parsing—there is no AST intermediate representation. As the parser recognizes language constructs, it calls code generation functions to emit YARV-style register-based bytecode instructions.

This design is possible because Lua's grammar is simple enough that no multi-pass analysis is needed. Forward references (like `goto` targeting a not-yet-defined label) are handled by patching bytecode after the fact.

### Core Parsing Functions

| Function | Role |
|----------|------|
| `chunk()` | Parses a block of statements |
| `statement()` | Dispatches to statement-specific parsers (`ifstat`, `whilestat`, `forstat`, `repeatstat`, etc.) |
| `expr()` | Entry point for expression parsing; delegates to `subexpr(0)` |
| `subexpr(limit)` | Precedence-climbing expression parser |
| `simpleexp()` | Parses atoms: literals, variables, constructors, function bodies |
| `suffixedexp()` | Handles postfix operations: `.field`, `[index]`, `:method()`, `(args)` |
| `primaryexp()` | Parses primary expressions (names and parenthesized expressions) |

### Operator Precedence

`subexpr()` implements precedence climbing using a static priority table:

```c
static const struct {
  lu_byte left, right;
} priority[] = {
  {10, 10}, {10, 10},           /* + - */
  {11, 11}, {11, 11},           /* * % */
  {14, 13},                     /* ^ (right-assoc) */
  {11, 11}, {11, 11},           /* / // */
  {6, 6}, {4, 4}, {5, 5},      /* & | ~ (bitwise) */
  {7, 7}, {7, 7},               /* << >> */
  {9, 8},                       /* .. (right-assoc) */
  {3, 3}, {3, 3}, {3, 3},      /* comparisons */
  {3, 3}, {3, 3}, {3, 3},
  {2, 2}, {1, 1},               /* and, or */
};
```

Right-associative operators (`^`, `..`) have `right < left` binding power, causing the recursive call to consume more tokens. Unary operators have a fixed priority of 12.

### Expression Description (expdesc)

Rather than building AST nodes, the parser uses `expdesc` structures to describe intermediate expression results:

- **`k`** (kind): Literal, variable, upvalue, table field, function call, etc.
- **`u`** (value): Numeric/string constant indices or register numbers.
- **`t`**, **`f`** (jump patches): Lists of unresolved conditional jumps, used for short-circuit evaluation of `and`/`or`.

Code generation functions take `expdesc` values and emit bytecode instructions, resolving register allocation and jump targets on the fly.

### Direct Code Generation

The parser calls code generation functions from `lcode.c` as it recognizes constructs:
- `luaK_codeABC()`, `luaK_codeABx()`: Emit instructions
- `luaK_exp2nextreg()`: Materialize an expression into the next available register
- `luaK_storevar()`: Generate code for variable assignment
- `luaK_concat()`, `luaK_goiftrue()`, `luaK_goiffalse()`: Handle conditional jumps and short-circuit logic

### Scope Management

- **`FuncState`**: Represents the function currently being compiled, including register allocation state, constant table, and upvalue list.
- **`BlockCnt`**: Tracks nested block scopes and local variable counts for proper cleanup (e.g., closing to-be-closed variables).
- **`Dyndata`**: Accumulates local variables, gotos, and labels during parsing.

## Go

### Design Philosophy

Go's grammar was intentionally designed to be parseable by a simple recursive descent parser with minimal lookahead. Rob Pike and the Go team made deliberate language design choices to keep the grammar simple:

- No operator overloading (fixed set of operators)
- No ternary operator
- Statement-oriented syntax (declarations use keywords like `var`, `func`, `type`)
- Automatic semicolon insertion at the scanner level (not the parser)
- No ambiguity between type expressions and value expressions
- Limited expression nesting in certain positions

### Parser Architecture

Go uses a hand-written recursive descent parser in both its standard library (`go/parser`) and compiler (`cmd/compile/internal/syntax`). Both operate on token streams produced by the corresponding scanner.

The compiler's parser is organized around top-level declarations (`importDecl`, `constDecl`, `typeDecl`, `varDecl`, `funcDecl`) with recursive descent into statements and expressions.

### Grammar Simplicity

Go's grammar is an LL(1) grammar by design—every parsing decision can be made by examining the next token. This eliminates the need for:
- Backtracking
- Extensive lookahead
- GLR or Earley parsing
- Parser-lexer feedback (except for semicolon insertion, which is handled purely in the scanner)

The grammar avoids ambiguities that plague other C-family languages:
- Type names and variable names are syntactically distinct (types start with uppercase or appear after keywords).
- Composite literal ambiguity (`if x == (T{})`) is resolved by requiring the type to be parenthesized in certain positions.
- No "most vexing parse" problem (C++'s `T x(args)` ambiguity between declaration and function call).

## Ruby

### The Bison/LALR Era

For its first 25+ years, CRuby used a Bison-generated LALR(1) parser. The grammar rules, lexer, and semantic actions all lived in the monolithic `parse.y` file (14,000+ lines).

#### LALR Parsing

LALR stands for "Look-Ahead LR parser." The algorithm:
1. Moves left to right through the token stream.
2. Uses bottom-up shift/reduce parsing.
3. Looks ahead one token to decide between shifting (pushing a new token) and reducing (applying a grammar rule).
4. Maintains a state table generated by Bison at build time.

Bison reads `parse.y` and produces `parse.c`, containing the LALR state tables and driver code. Grammar rules use BNF notation with Bison-specific directives (`$$`, `$1`, `$2`, etc.) for passing values between parent and child rules.

#### Why Ruby's Grammar Is Hard

Ruby's grammar is deeply context-sensitive, making it a poor fit for LALR:

- **Operator ambiguity**: `*` means multiplication or splat depending on spacing and context.
- **`...`**: Range operator or argument forwarding, depending on position.
- **`do`**: Block opener in different syntactic roles.
- **Newline sensitivity**: Newlines are statement terminators, but only sometimes. Extensive state tracking determines whether a newline should be honored based on context (hash values vs. method parameters).
- **Local variable influence**: `a /b#/` parses as either method call with regex argument or variable division, depending on whether `a` is a declared local variable.
- **Regex termination**: `%r{foo {}}` requires understanding balanced braces within regex delimiters.

These complexities forced CRuby to embed extensive lexer state management (the `lex_state` machine) alongside the Bison grammar, resulting in an interleaving of parser and lexer logic that was extremely difficult to maintain.

#### Lrama Replacement

In Ruby 3.3 (2023), Kaneko Yuichiro replaced Bison with Lrama, a reimplementation of Bison's LALR parser generator written in pure Ruby. Lrama reads the same `parse.y` and generates the same `parse.c`, but being written in Ruby, it's easier for the Ruby core team to modify and extend. Lrama adds features Bison lacks, like parameterized rules.

### The Prism Era

Shopify developed Prism (originally YARP), a hand-written recursive descent parser in C that replaces the Bison-generated parser entirely. Prism ships as a library starting with Ruby 3.3 and is becoming the default parser.

#### Motivations

1. **Error tolerance**: The Bison parser stopped at the first syntax error. Prism continues past errors using multiple recovery strategies.
2. **Portability**: Prism has zero dependencies on CRuby internals. JRuby and TruffleRuby use it directly.
3. **Performance**: Hand-written parsers allow compiler optimizations that generated jump tables prevent. Prism parses ~50,000 files in 4.49s with 10.94 MiB peak memory.
4. **Standardized AST**: Prism defines a new AST format designed in consultation with JRuby, TruffleRuby, and tool maintainers, replacing CRuby's ad-hoc tree that included parser-specific nodes like `vcall`.

#### Error Recovery Strategies

Prism implements four recovery techniques:

1. **Missing token insertion**: When `end` or other keywords are absent, the parser inserts them and continues.
2. **Missing node insertion**: Incomplete expressions get placeholder nodes (e.g., missing right-hand side of `+`).
3. **Token skipping**: Unrecognizable tokens in context are bypassed.
4. **Context-based recovery**: The parser analyzes surrounding context to determine the best recovery action (e.g., `)` closing an argument list terminates a malformed expression).

#### APIs

Prism provides two access methods:
- **C API**: Direct function calls for languages with FFI (Rust, Zig, etc.).
- **Serialization API**: Parse in C, serialize to binary, deserialize in target language. Deserialization is ~10x faster than parsing, enabling pre-serialized standard libraries for faster boot times.

## Parsing Algorithms

### Recursive Descent

A top-down parsing technique where each grammar rule becomes a function. The parser starts from the start symbol and recursively expands non-terminals.

**Grammar class**: Not formally related to CFGs—accepts a different set of languages depending on implementation choices (backtracking, lookahead). Typically targets LL(k) grammars.

**Translation from grammar to code:**
- Terminals -> token matching and consumption
- Non-terminals -> function calls
- Alternation (`|`) -> `if`/`switch` statements
- Repetition (`*`, `+`) -> `while`/`for` loops
- Optional (`?`) -> `if` statements

**Advantages:**
- Maximum flexibility: any context-sensitive hack can be injected
- Maximum performance: hand-tuned hot paths
- Excellent error reporting possible (Rust's `rustc` is the gold standard)
- Easy to understand: code mirrors grammar structure

**Disadvantages:**
- Cannot handle left recursion without transformation
- Silently ignores ambiguities (no static analysis of the grammar)
- Operator precedence requires careful manual encoding (or Pratt parsing)
- Substantial boilerplate for large grammars

**Used by**: Rust, Go, JavaScript (V8, Acorn, Babel), Ruby (Prism), GCC, Clang, Swift, many others.

### Pratt Parsing / Precedence Climbing

A technique for parsing expressions with operator precedence within a recursive descent framework. See the [deep dive section](#pratt-parsing-deep-dive) below.

### PEG (Parsing Expression Grammars)

PEG parsers use ordered choice (`/`) instead of unordered alternation (`|`). The first matching alternative always wins, eliminating ambiguity by definition.

**Grammar class**: Disjoint from CFGs. Every PEG grammar is unambiguous, but some unambiguous CFGs cannot be expressed as PEGs, and some PEG languages are not context-free.

**Advantages:**
- Unambiguous by construction
- Supports unlimited lookahead via packrat memoization
- Clean, readable grammar specifications
- Can eliminate the separate lexer (scannerless parsing)

**Disadvantages:**
- Ordered choice can silently hide bugs (an early alternative may consume input that a later, correct alternative should have matched)
- No static ambiguity detection (same fundamental blindness as recursive descent)
- Left recursion requires special handling (CPython's pegen solves this via memoization tricks)
- Packrat memoization trades memory for guaranteed O(n) time

**Used by**: Python 3.9+ (pegen), Pest (Rust), PEG.js/Peggy (JavaScript), LPEG (Lua).

### LL Parsing

Top-down parsing using a parse table computed from the grammar. LL(k) uses k tokens of lookahead.

**Grammar class**: Strict subset of LR; LL(1) is the most restrictive practical class. Forbids left recursion and requires unique first-set selection.

**Advantages:**
- Predictive (no backtracking needed for LL(1))
- Natural mapping to recursive descent implementation
- Static grammar analysis catches some ambiguities
- Generated parsers can ensure recursive descent doesn't have hidden ambiguities

**Disadvantages:**
- No left recursion
- Many natural grammar constructs require awkward transformations
- LL(1) conflicts are common and hard to resolve

**Used by**: Python (pre-3.9), ANTLR (LL(*) with arbitrary lookahead).

### LR Parsing (SLR, LALR, Canonical LR)

Bottom-up parsing using shift/reduce operations driven by a state table computed at generator time.

**Grammar class**: LR(1) is the largest practical subset of unambiguous CFGs. LALR(1) is a slightly smaller subset that produces more compact tables. SLR is smaller still.

**Advantages:**
- Statically proves grammar unambiguity at generator time
- Handles left recursion naturally
- Grammar notation close to mathematical CFG notation
- Performance: O(n) parsing, thousands of lines per second

**Disadvantages:**
- Shift/reduce and reduce/reduce conflicts can be cryptic to debug
- Error recovery is historically poor (Yacc/Bison's `error` token produces frustrating results)
- Grammar must be refactored to fit LALR(1) constraints
- Right recursion in expressions requires explicit precedence declarations

**Used by**: Ruby (CRuby via Bison/Lrama), PostgreSQL (Bison), PHP (Bison), MySQL.

### GLR (Generalized LR)

Extension of LR parsing that handles ambiguous grammars by maintaining multiple parse stacks simultaneously. When a conflict is encountered, the parser forks into parallel stacks.

**Grammar class**: All context-free grammars, including ambiguous ones.

**Advantages:**
- Parses any CFG
- Identifies all ambiguities at runtime (produces parse forests)
- Can be combined with incremental parsing (tree-sitter)
- Practical performance for real grammars (ambiguity forks are rare in practice)

**Disadvantages:**
- O(n^3) worst case (n^4 in some formulations)
- Ambiguities discovered only at runtime, not statically
- Disambiguation requires precedence/associativity annotations
- More complex implementation than plain LR

**Used by**: Tree-sitter, Elkhound, Bison (GLR mode).

### GLL (Generalized LL)

Top-down analogue of GLR. Uses a graph-structured stack to explore all possible LL derivations simultaneously.

**Grammar class**: All context-free grammars.

**Advantages:**
- Handles any CFG with top-down control flow
- Produces parse forests for ambiguous grammars
- O(n^3) worst case (better than GLR's O(n^4))
- Natural fit for grammar extension and composition

**Disadvantages:**
- Less mature than GLR
- Runtime ambiguity detection only
- Higher constant factors than simple recursive descent

### Earley Parsing

A chart-based algorithm that maintains sets of "items" (partially completed grammar rules) at each input position.

**Grammar class**: All context-free grammars.

**Advantages:**
- Parses any CFG
- O(n) for unambiguous grammars, O(n^2) for unambiguous grammars with some structural properties, O(n^3) worst case
- Good theoretical properties
- Natural for natural language processing

**Disadvantages:**
- Higher constant factors than LR or recursive descent
- Less practical for programming language parsers
- Ambiguity handling through parse forests can be complex
- Few production-quality implementations

**Used by**: Marpa (Perl), some NLP systems.

### Parser Combinators

A functional programming approach where parsers are functions that can be combined using higher-order functions (sequence, choice, repetition, etc.).

**Grammar class**: Typically PEG-like (ordered choice, recursive descent semantics).

**Advantages:**
- Grammar is expressed in the host language (no separate grammar file)
- Composable: parsers can be built modularly and reused
- Good for embedding DSL parsers
- Type system can enforce structural constraints

**Disadvantages:**
- Same ambiguity blindness as recursive descent/PEG
- Performance overhead from function call dispatch and closure allocation
- Error messages tend to be poor without careful engineering
- Backtracking can cause exponential blowup without memoization

**Used by**: Nom (Rust), Parsec/Megaparsec (Haskell), FParsec (F#), Combine (Swift), Chumsky (Rust).

### Algorithm Comparison Summary

| Algorithm | Grammar Class | Time Complexity | Ambiguity Detection | Error Recovery | Typical Use |
|-----------|--------------|-----------------|---------------------|----------------|-------------|
| Recursive Descent | LL-ish (ad hoc) | O(n) typical | None | Excellent (if hand-tuned) | Production compilers |
| Pratt/Precedence Climbing | Operator expressions | O(n) | None | Good | Expression parsing within RD |
| PEG (packrat) | PEG | O(n) guaranteed | None (by construction) | Moderate | Python 3.9+, tooling |
| LL(1) | LL(1) | O(n) | Static (partial) | Good | ANTLR, old Python |
| LALR(1) | LALR(1) | O(n) | Static (full within class) | Poor (traditionally) | Ruby, PostgreSQL |
| GLR | All CFGs | O(n^3) worst | Runtime | Good | Tree-sitter |
| GLL | All CFGs | O(n^3) worst | Runtime | Moderate | Research |
| Earley | All CFGs | O(n^3) worst | Runtime | Moderate | NLP |
| Combinators | PEG-like | O(n) with memo | None | Variable | Embedded DSLs |

### Laurence Tratt's Recommendation

From the authoritative "Which Parsing Approach?" survey:

> "LR parsing is the best approach for the vast majority of purposes: strongest practical safety guarantees, allows good grammar readability, and has decent performance."

Exceptions: LL + hand-written recursive descent for highest performance needs; generalized parsing for grammar composition requirements.

## Error Recovery

### Panic Mode

The simplest and most widely used recovery strategy. When the parser encounters an error:

1. **Report the error** with the current token's location and context.
2. **Enter panic mode**: Mark the parser as being in an error state to suppress cascading error messages.
3. **Discard tokens** until reaching a **synchronization point** — a token that is likely to begin a new, valid construct.
4. **Reset parser state**: Unwind the call stack (in recursive descent) or reset the parse table state (in LR) to the synchronization point.
5. **Resume parsing** from the synchronization token.

**Synchronization tokens** are typically:
- Statement terminators: `;`, newline
- Block delimiters: `}`, `end`
- Statement-starting keywords: `if`, `while`, `for`, `return`, `class`, `fn`

The effectiveness of panic mode depends heavily on the choice of synchronization tokens. Good choices recover quickly from common errors. Poor choices skip too much valid code.

**Implementation in recursive descent** (from Crafting Interpreters): The `synchronize()` method uses Java's exception mechanism to unwind the call stack, then discards tokens until finding a statement boundary. The `ParseError` exception serves as the panic-mode trigger.

**Advantages**: Simple to implement, works with any parsing method, prevents cascading errors.
**Disadvantages**: Skips potentially valid code without checking it for additional errors. Loses diagnostic quality for the skipped region.

### Error Productions

Augment the grammar with rules that match known-invalid syntax, allowing the parser to continue while reporting an error.

Examples:
- A rule matching `if expr { }` without parentheses in a C-like language, producing a "missing parentheses around if condition" error.
- A rule matching `x = ;` (assignment with missing right-hand side), producing a "missing expression" error.

**Advantages**: Produces highly specific, actionable error messages. The parser maintains full context through the error.
**Disadvantages**: Accumulates over time ("like barnacles on a ship"). Each common mistake requires a dedicated grammar rule. Can make the grammar harder to understand.

### Token Insertion / Deletion

When the parser expects token T but sees token U:

- **Insertion**: Synthesize a token T with a dummy value, report the error ("expected T"), and continue parsing as if T had been present. Used by Prism (Ruby) for missing `end` keywords.
- **Deletion**: Skip token U, report the error ("unexpected U"), and retry. Used when a stray token appears in an otherwise valid context.

These strategies work best for single-token errors (typos, forgotten semicolons) and can cascade badly for larger structural errors.

### Context-Based Recovery

The parser uses its current context (which grammar rules are active on the stack) to make informed recovery decisions. For example:

- Inside a function call's argument list, encountering `)` terminates the list even if the last argument is malformed.
- Inside a block, encountering `}` or `end` closes the block.
- Inside an array literal, encountering `]` closes the array.

This is essentially panic mode with per-context synchronization token sets. It produces better recovery than a single global set of synchronization tokens.

### Rust's Approach to Error Quality

The Rust compiler achieves its high-quality error messages through:

1. **Superset parsing**: Accepting more than valid Rust and diagnosing the excess.
2. **Pattern matching on common mistakes**: Detecting `=` where `==` was intended, `function` instead of `fn`, C-style casts, etc.
3. **Contextual messages**: "Did you mean..." suggestions based on the parsing context.
4. **Unicode homoglyph detection**: Mapping visually similar Unicode characters to ASCII equivalents.
5. **Span-rich diagnostics**: Every error message includes source spans, and many include secondary spans showing related code.

### Modern Best Practices

1. **Always recover**: Never stop at the first error. Users expect to see all errors.
2. **Suppress cascading errors**: After a recovery, suppress further errors until parsing is clearly back on track (e.g., after successfully parsing a complete statement).
3. **Preserve partial AST**: Store error nodes in the AST so downstream tools (IDEs, linters) can work with incomplete code.
4. **Use context for recovery**: Choose synchronization tokens based on what grammar rules are currently active.
5. **Provide suggestions**: When the parser can guess what the user intended, suggest the fix.

## Incremental Parsing

### The Problem

Traditional parsers reparse the entire source file on every change. For large files in IDEs, this causes noticeable lag in syntax highlighting, error reporting, and code navigation. Incremental parsing solves this by reusing unchanged portions of the previous parse tree.

### Tree-sitter

Tree-sitter is the dominant incremental parsing system, used by Neovim, Helix, Zed, Emacs, and many other editors.

#### Algorithm

Tree-sitter uses a **GLR-based incremental LR parsing** algorithm:

1. **Parse table generation**: At grammar-author time, tree-sitter generates LR parse tables from a grammar specification written in JavaScript. The grammar is processed through conflict detection, and ambiguities are resolved via precedence annotations or deferred to runtime GLR handling.

2. **Initial parse**: The first parse produces a concrete syntax tree (CST) where every node corresponds to a grammar symbol. Nodes store byte ranges and point coordinates (row/column).

3. **Incremental reparse**: When the source is edited:
   - The editor provides the edit description (`TSInputEdit`: start byte, old end byte, new end byte, plus row/column coordinates).
   - The parser traverses the old tree via a `ReusableNode` iterator.
   - At each position, it checks whether a subtree from the old tree can be reused:
     - The byte position must match the current parsing position.
     - No edits must fall within the node's byte range.
     - The external scanner state must be compatible.
     - The node's first leaf symbol must be valid in the current parser state.
   - Reusable subtrees are incorporated directly into the new tree, sharing memory with the old tree.
   - Only the regions affected by the edit are actually re-parsed.

#### GLR Stack Management

Tree-sitter maintains multiple parse stacks to handle ambiguity:

- **Up to 6 concurrent versions** (with temporary overflow to 10).
- Each `StackHead` has its own parser state.
- Nodes form a directed acyclic graph (shared prefixes across versions).
- Versions are pruned based on cost: non-error versions are preferred, lower error counts win, higher dynamic precedence breaks ties.
- Versions merge when they reach the same parser state at the same byte position with compatible error costs.

#### Error Recovery

Tree-sitter's error recovery is designed for robustness in the face of incomplete or invalid code (which is the normal state of code being edited):

- When no valid action exists, the parser enters recovery mode.
- It attempts to skip tokens until finding a valid continuation.
- Error nodes are inserted into the CST to represent invalid regions.
- Recovery respects context (blocks, delimiters) when possible.
- The parser continues past errors, producing a complete tree for the entire file.

#### External Scanners

For context-sensitive tokens that LR parse tables cannot handle (Python indentation, Ruby heredocs, template strings), tree-sitter supports external scanners:

- Custom C/C++ functions that can maintain arbitrary state.
- Scanner state is serialized/deserialized at parse checkpoints for incremental parsing compatibility.
- Called via function pointers (`create`, `scan`, `serialize`, `deserialize`, `destroy`).

#### Performance

- The old tree and new tree share unchanged subtrees (structure sharing), minimizing memory allocation.
- A single-token cache stores `(byte_index, scanner_state)` pairs to avoid redundant lexing during backtracking.
- Every 100 parse actions, a cancellation flag is checked for timeout management.
- Parsing is fast enough to run on every keystroke.

#### Changed Range Computation

`ts_subtree_get_changed_ranges()` computes the set of byte ranges where the old and new trees differ, enabling applications to update only the affected regions (e.g., re-highlighting only changed syntax).

### Other Incremental Approaches

**Lezer** (CodeMirror): An incremental parser used by the CodeMirror editor. Uses an LR-based approach similar to tree-sitter but designed specifically for the web (JavaScript/TypeScript). Grammars are specified in a custom format. Produces immutable persistent trees with structure sharing.

**Roslyn** (C#): Microsoft's C# compiler uses incremental parsing where the parser tracks which syntax nodes are affected by an edit and rebuilds only those nodes. The immutable red-green tree architecture allows sharing unchanged nodes between parse tree versions.

**IntelliJ Platform**: Uses a "chameleon" node approach where parts of the parse tree are lazily reparsed. Nodes can be in a "not yet parsed" state and are expanded on demand.

## Pratt Parsing Deep Dive

### Historical Context

Vaughan Pratt presented "Top Down Operator Precedence" at the first ACM Symposium on Principles of Programming Languages in Boston, 1973. He designed it for the CGOL programming language and described it as combining "the best properties of Recursive Descent and Floyd's Operator Precedence" while being "simple to understand, trivial to implement, easy to use, extremely efficient, and very flexible."

The algorithm languished in relative obscurity for decades until Douglas Crockford popularized it in his 2007 article "Top Down Operator Precedence" (using JavaScript), and Bob Nystrom further clarified it in his 2011 article "Pratt Parsers: Expression Parsing Made Easy."

### Core Concepts

#### Binding Power

Traditional parser literature talks about "precedence levels." Pratt parsing reframes this as **binding power** — the strength an operator exerts to hold onto the operands around it.

In `A + B * C`, the `*` operator has higher binding power than `+`, so it "grabs" `B` more tightly. The expression parses as `A + (B * C)`.

Each operator has two binding power values:
- **Left binding power (lbp)**: How strongly the operator attracts the operand to its left.
- **Right binding power (rbp)**: How strongly the operator attracts the operand to its right.

#### Nud and Led

Pratt's original terminology:
- **nud** ("null denotation"): How a token behaves when it appears at the **beginning** of an expression (prefix position). Used by literals, variables, unary operators, and opening delimiters.
- **led** ("left denotation"): How a token behaves when it appears **after** a left operand (infix/postfix position). Used by binary operators, postfix operators, and function call syntax.

In modern implementations, these are often called `prefix` and `infix` (or implemented as `PrefixParselet` and `InfixParselet` objects).

### The Algorithm

The core is a single function, typically called `expr_bp` or `expression`:

```
function expression(min_bp):
    # Parse prefix
    token = advance()
    lhs = token.nud()

    # Parse infix/postfix
    while peek().lbp > min_bp:
        token = advance()
        lhs = token.led(lhs)

    return lhs
```

The function:
1. Consumes a token and processes it as a prefix (via `nud`).
2. Loops while the next token's left binding power exceeds the current minimum.
3. In each iteration, consumes the operator and processes it as infix (via `led`), passing the left operand.
4. Returns when encountering an operator too weak to continue.

### Handling Operator Types

#### Prefix Operators (Unary)

A prefix operator has only a right binding power. Its nud method recursively calls `expression(rbp)` to parse its operand:

```
nud for '-':
    operand = expression(PREFIX_BP)
    return UnaryExpr('-', operand)
```

The prefix binding power is typically set high (e.g., 100) so unary operators bind tightly.

#### Infix Operators (Binary)

An infix operator has both left and right binding powers. Its led method takes the left operand and recursively parses the right:

```
led for '+':
    right = expression(rbp)      # rbp = lbp + 1 for left-assoc
    return BinaryExpr('+', left, right)
```

#### Postfix Operators

A postfix operator has only a left binding power. Its led method takes the left operand and returns without parsing further:

```
led for '!':   # factorial
    return PostfixExpr('!', left)
```

#### Mixfix Operators

Complex operators like the ternary `? :` are implemented as infix operators whose led method parses additional tokens:

```
led for '?':
    then_expr = expression(0)     # Parse between ? and :
    consume(':')
    else_expr = expression(rbp)   # Parse after :
    return TernaryExpr(left, then_expr, else_expr)
```

Indexing (`a[i]`) is implemented similarly as a postfix/infix operator whose led method parses the index expression and consumes `]`.

### Associativity Through Asymmetric Binding Powers

The key insight: **associativity is encoded by the difference between left and right binding powers.**

For an operator at precedence level P:

| Associativity | Left BP | Right BP | Effect |
|---------------|---------|----------|--------|
| Left | P | P + 1 | Right operand binds tighter, preventing same-precedence operators from being consumed on the right |
| Right | P | P | Right operand at same level, allowing same-precedence operators to be consumed on the right |
| Non-associative | P | P + 1 | Same as left-assoc, but add a check that rejects a second operator at the same level |

Example with left-associative `+` (lbp=1, rbp=2):
- `1 + 2 + 3`: After parsing `1 + 2`, the second `+` has lbp=1, which is NOT > min_bp=2, so the loop stops. The first `+` returns `(1+2)`, and the outer call handles `(1+2) + 3`.

Example with right-associative `^` (lbp=2, rbp=1):
- `2 ^ 3 ^ 4`: After parsing `2`, the first `^` calls `expression(1)`. The second `^` has lbp=2 > min_bp=1, so it's consumed recursively, producing `2 ^ (3 ^ 4)`.

### Equivalence to Precedence Climbing

Andy Chu (Oil Shell blog) demonstrated that Pratt parsing and precedence climbing are the same algorithm, with different presentations:

- Precedence climbing uses a numeric `min_prec` parameter and switches on operator type.
- Pratt parsing uses binding power values and dispatches through method/function tables.

The transformation from precedence climbing to Pratt parsing involves:
1. Replace the `min_prec` parameter with `rbp` (right binding power).
2. Replace the operator type switch with a `led` dispatch table.
3. Replace the prefix handling switch with a `nud` dispatch table.
4. Replace precedence + associativity pairs with asymmetric binding powers.

The algorithmic core is identical: a loop that consumes operators while their precedence exceeds a threshold, with recursive calls using an adjusted threshold for associativity.

The key contribution of Pratt's formulation over precedence climbing is **modularity**: adding a new operator requires only adding an entry to the dispatch table, not modifying the core parsing loop.

### Where Languages Use Pratt Parsing

| Language/Tool | Context |
|--------------|---------|
| Rust (rustc) | Expression parsing within the recursive descent parser |
| Lua | `subexpr()` uses a precedence table (effectively Pratt) |
| JavaScript (V8) | Expression parsing |
| Python (CPython) | Expression precedence in the PEG grammar |
| Go | Expression parsing |
| Clang | C/C++/ObjC expression parsing |
| Ruby (Prism) | Expression parsing |

Nearly every hand-written recursive descent parser uses some form of Pratt parsing or precedence climbing for expression parsing, even if the rest of the grammar is handled by recursive descent functions.

### Implementation Patterns

**Table-driven**: Store `(nud, led, lbp)` in a lookup table indexed by token type. Maximum extensibility; adding operators means adding table entries.

**Switch-based**: Use a switch statement in `expr_bp` to dispatch prefix and infix handling. Simpler for fixed grammars; all precedence logic is visible in one place.

**Parselet objects** (Bob Nystrom's approach): Define `PrefixParselet` and `InfixParselet` interfaces. Register parselet instances for each token type. Maximum OOP modularity but more allocation overhead.

**Binding power function**: A function that maps token types to `(left_bp, right_bp)` pairs. The simplest approach for languages with a fixed operator set.
